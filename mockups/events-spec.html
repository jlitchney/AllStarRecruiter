Roster Status Lifecycle

Statuses flow in one direction (no going back to Invited once an outcome is recorded). Each transition applies the corresponding Association Tag to the candidate.

Invited Confirmed or Declined Attended or No-Show or Rescheduled
  • When a candidate is added to the roster, they receive Invited status and the tag_invite_id tag is applied (if set).
  • Self-registrations via the public registration page also start as Invited.
  • Post-event outcomes (Attended / No-Show / Rescheduled) apply the corresponding tag, which fires any workflow automations listening for that tag.
  • Bulk status updates must apply tags to all affected candidates atomically — either all succeed or roll back.

Registration Page (Group Events Only)
  • Toggle to enable/disable a public-facing registration page.
  • Two source modes:
    • Web-builder page — Dropdown of pages already created in the web-builder. Stores the web-builder page ID in reg_page_ref.
    • Custom URL — Free-text URL input. Stores raw URL in reg_page_ref.
  • The resolved public URL (reg_page_url) and QR code (reg_page_qr_url) are generated server-side on save — not available on the create form.
  • After save, the URL and QR code are shown on the Event Detail page with Copy and Download actions.
  • Disabling the registration page after it has been published should show a confirm dialog warning that the public URL will stop accepting registrations.
  • Self-registrations via the public page POST to the event roster with source: self_registered and status Invited, and apply tag_invite_id if set.
  • The public registration page is served by the All-Star Events web app (events-six-orpin.vercel.app).
Association Tags

Four tag slots, one per roster outcome. Each shows a dropdown of existing tags plus an always-visible inline form to create a new tag.

Tag SlotWhen Applied
Invite TagWhen a candidate is added to the roster (manually, by tag, or via self-registration)
Attended TagWhen roster status is set to Attended
No-Show TagWhen roster status is set to No-Show
Reschedule TagWhen roster status is set to Rescheduled
  • Create-new tag inline form: Group selector (existing tag groups for the org) + Tag name input + Create button. Always visible — no hidden panel.
  • On "Create", POST the new tag, then immediately select it in the dropdown above.
  • Tags trigger workflow automations. These connections are configured outside this form (in the workflow builder).
  • Group scope Invite Tag: a live candidate count should show below the dropdown once a tag is selected (e.g. "47 candidates will be added to this event").
  • On form submit, if a group Invite Tag is set, the back-end queries all candidates with that tag and bulk-inserts them into EventRoster immediately — not lazily.
UTM Tracking
  • Three fields: utm_source, utm_medium, utm_campaign. Free text, stored on the Event record.
  • These values should be appended as query parameters to the registration page URL when generating reg_page_url.
  • Visible and editable in Association Tags view on the Event Detail page.

Page 4 — Event Types (Settings)

Event Types — Settings Page

Route: /settings/event-types  ·  Open mockup →
Purpose

Admins manage the custom event types available for their department. Types drive the color coding on calendars, list views, and candidate timelines.

Add / Edit Form (Left Panel)
  • Fields: Event Name, Color (palette picker, stored as hex), Active toggle.
  • Clicking "Edit" on a table row populates the left form with that row's values and changes the Save button label to "Update Event Type".
  • "Clear" resets the form to blank / default state.
Type Table (Right Panel)
  • Columns: Name (with color dot), Status (Active/Inactive badge), Actions (Edit / Delete).
  • Header badge shows count of active types.
  • Delete must be blocked if any Event record currently references that type. Show an error: "This type is used by N event(s) and cannot be deleted. Set it to Inactive to hide it from new events."
  • Inactive types are hidden from the Event Type dropdown on the Add Event form but all historical event records referencing them are preserved.
Scoping

Types are scoped to org_id. Different departments have entirely separate type lists. This page only shows and manages types for the logged-in organization.

API Endpoints
GET
/api/event-types?org_id=
All types (active + inactive) for this org.
POST
/api/event-types
Create new type. Body: { org_id, name, color, is_active }.
PUT

Update name, color, or active state.
DELETE
/api/event-types/{id}

Cross-Cutting Concerns

Multi-Tenancy
  • All records (events, types, templates, roster rows) are scoped to org_id. Never query across orgs.
  • All API routes must validate that the requesting user belongs to the org_id in the request.
Timezones
  • All datetimes stored as UTC. Display in the logged-in user's local timezone.
  • Date groupings (Today / Upcoming / Past) computed server-side in UTC — no client-side date math for grouping.
Workflow / Tag Integration
  • Applying a tag to a candidate (via any roster status change) must fire the tag-applied event in the workflow engine. This is the primary automation trigger — keep it synchronous enough to be reliable but don't block the UI response.
  • The four Association Tags (Invite, Attended, No-Show, Reschedule) are the only event-to-tag bridge. No other implicit tags are applied by this module.
Public Registration App
  • Served separately at events-six-orpin.vercel.app (All-Star Events web app).
  • Self-registrations POST to /api/events/{id}/roster with source: self_registered. The CRM back-end handles tag application identically to manual adds.
  • UTM parameters from the event record are appended to the public URL at generation time.
Rich Text
  • Description field uses a rich text editor (e.g. Tiptap or Quill). Store and render as sanitized HTML.
  • Inline edit on Event Detail page uses contenteditable — apply the same sanitization on save.
